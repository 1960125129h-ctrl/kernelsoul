Kernelsoul

AI 角色扮演的确定性角色引擎
在 LLM 与角色卡之间插入持久化状态层、规则执行层和语义渲染管线——让角色卡从"被阅读的故事"升级为"被执行的程序"。

Kernelsoul（原名 Kernelsoul）是一个面向 AI 角色扮演的开源角色引擎。它不是又一个"前端酒馆"——而是一个位于 LLM 下游的角色操作系统，负责维护角色状态的持久性、执行行为规则的确定性、以及叙事输出的语义表现力。

为什么需要 Kernelsoul？

传统的 AI 角色扮演流水线非常简单：

角色卡 JSON  →  Prompt 组装  →  LLM  →  输出文本

这个模式有三个根本问题：

❌ 角色没有"状态"

角色卡是一个静态 JSON 文件。血量、好感度、背包物品——这些在每次对话后都会消失，除非 LLM 自己记在对话历史里（LLM 显然不擅长精确记数字）。结果是角色行为前后不一致。

❌ 规则靠"概率"执行

"不能出戏"、"保持冷酷"——这些约束写在 character note 里，但 LLM 不一定遵守。10 次对话里大概有 2-3 次它会忽略。

❌ 数值状态直接"裸奔"到 prompt

"HP: -12"、"mood: 3"这种干瘪数字直接塞进 prompt。缺乏叙事表现力，LLM 被迫自己猜"HP -12 该怎么描述"。

Kernelsoul 在每个环节插入了专门的中间层来系统性地解决这些问题。

核心特性

1. 强类型游戏状态（GameState）

角色不再是平面 JSON，而是拥有运行时状态的对象。变量有类型约束和读写权限：

字段类型AI 可写初始值说明
hpint✅150生命值
max_hpint❌150只读上限
moodint✅0-5 ～ 5
energyint✅100 ～ 10
is_alivebool⚠️True系统保护
inventorylist[str]✅[]背包物品
goldint✅100金钱

AI 在每次对话响应中通过 JSON 字段 state_changes 修改状态。引擎强制执行范围约束和白名单——不在白名单里的字段 AI 无法写入。

2. 确定性 DSL 规则引擎

Kernelsoul 定义了自己的领域特定语言（SLL —— Kernelsoul Logic Language），专门用于描述角色行为规则。规则由 SLL 编译器（递归下降解析器）解析为可执行规则集：

WHEN hp < 30：
    IF has_potion：
        USE potion
        THEN "她颤抖着摸出一瓶红色药水，用牙齿咬开瓶塞一饮而尽。伤口在缓缓愈合。"
        BECAUSE "角色处于危险时会使用随身携带的治疗道具"
    ELSE：
        THEN "她捂住伤口，咬紧牙关。血从指缝间渗出来。"
        BECAUSE "角色没有治疗资源时会硬撑"

WHEN is_alive = False：
    THEN "角色已经无法继续行动了"
    BECAUSE "死亡状态"
    BLOCK 所有非检查行动

规则引擎在每次回合结束后自动检查所有 WHEN 条件，命中即执行对应的行为——这是确定性的，不依赖 LLM 概率性理解。

3. 语义叙事渲染引擎（行业首创）

传统的做法是把数值状态直接塞进 prompt："mood: -2, energy: 8"。Kernelsoul 的语义渲染器将其转化为叙事化的状态描述——让 LLM 感受到的是"氛围"而不是"数字"。

采用三模式设计：

模式行为适用场景
direct直出数值：HP: 35/150调试 / 简洁模式
semantic叙事描述：状态大致良好但已显疲态沉浸式 RP
hybrid数值 + 描述混合既要可读性又要精确性

底层通过语义轴（将游戏变量映射到高维语义空间）、Warp 函数（非线性映射）、Gram 矩阵解耦（分离正交维度）和语义动量（SLERP 插值）实现叙事感知。

语义轴构建 低维子空间投影 Warp 函数标定 SLERP 动量插值

4. 三层分级记忆

记忆不是简单地把对话拼接起来——而是按时间跨度和重要性分三层管理：

层级轮数摘要上限触发条件
轻量10 轮200 字每 5 轮打包
深度50 轮2500 字8 条轻量 → 合并
史诗200 轮5000 字32 条深度 → 合并

每次对话引擎只注入当前所需的上下文量：近 10 轮完整记录 + 轻量摘要 + 上一级深度摘要。全程用 LLM 压缩摘要，保持关键信息不丢失。

5. 决策溯源

每个决策都有记录。"刚才这个角色为什么突然怒了？"——查询决策日志即可：

WHY 角色行为: 
  → 规则: WHEN mood < -3 → 敌对态度
  → 状态: mood -5 (hp损失+连续受挫)
  → 触发: 玩家再次挑衅
  → 结果: 进入敌对模式, 拔剑

这是 Kernelsoul 独有的能力——任何竞品都无法追溯 AI 角色行为的原因。

6. 多草稿与版本管理

每次用户输入后，引擎可以生成多个候选回复草稿（/roll），用户预览后选择最满意的一条（/select）。可以收藏重要节点，也可以回滚到之前的会话版本。这解决了"一次对话决定永久命运"的问题。

7. 角色卡兼容

支持 chara_card_v4 chara_card_v2 格式，兼容现有 SillyTavern 生态的角色卡。角色卡中嵌入 DSL 源码字段（character_rules_dsl），让普通角色卡也能享受确定性规则。

8. 沙箱插件系统

插件有完整的生命周期钩子（init/before_turn/after_turn/cleanup）、权限审查（高风险操作日志记录）、沙箱隔离（异常不影响引擎）。manifest.json 规范注册，支持动态加载/卸载。

与竞品的核心差异

Kernelsoul 不是又一个"SillyTavern 替代品"——它是一个完全不同层级的项目。竞品解决的是"怎么聊"，Kernelsoul 解决的是"角色怎么像角色"。

维度KernelsoulSillyTavern
定位后端角色引擎前端酒馆
角色状态强类型 GameState，AI 可控白名单无状态，角色是静态 JSON
规则执行确定性 DSL 引擎，回合级条件判断纯 prompt 暗示，不可靠
叙事渲染语义轴/Warp/动量/Gram矩阵纯数值直出
DSL 语言SLL 递归下降解析器无
决策溯源WHY 查询、决策日志黑盒 LLM 输出
记忆管理三层分级压缩（10/50/200轮）简单摘要 / 向量扩展
草稿管理多草稿 + 收藏 + 版本回滚无
角色格式V4 标准 + DSL 源码chara_card_v3（事实标准）
集成方式gRPC API / pip 包 / ST 插件独立前端
