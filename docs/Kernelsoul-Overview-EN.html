Kernelsoul (formerly V4 Character OS) is an open-source AI character roleplay engine. It is not "yet another chat frontend" - it is a character operating system that lives downstream of the LLM, responsible for maintaining character state persistence, deterministic behavior rule execution, and semantically expressive narrative output.

Why Kernelsoul?

Traditional AI character roleplay pipelines look like this:

Character Card JSON → Prompt Assembly → LLM → Text Output

This model has three fundamental problems:

No "State" for Characters

A character card is a static JSON file. HP, affinity, inventory items - all disappear after each conversation unless the LLM remembers them in the chat history (which LLMs are terrible at). The result is inconsistent character behavior.

Rules Are "Probabilistic"

"Stay in character" or "remain cold" - these constraints live in character notes, but the LLM may or may not follow them. Roughly 20-30% of the time, it will ignore them.

Naked Numeric State in Prompts

"HP: -12", "mood: 3" - these dry numbers are stuffed directly into the prompt without any narrative framing. The LLM must guess what "HP -12" looks like in story terms.

Kernelsoul inserts dedicated middleware layers at every stage to solve these problems systematically.

Core Features

1. Strongly-Typed GameState

Characters are no longer flat JSON objects - they have runtime state objects with type constraints and read/write permissions:

FieldTypeAI WritableDefaultDescription
hpint✅150Hit points
max_hpint❌150Read-only cap
moodint✅0-5 to 5
energyint✅100 to 10
is_alivebool⚠️TrueSystem protected
inventorylist[str]✅[]Item bag

2. Deterministic DSL Rule Engine

Kernelsoul defines its own domain-specific language - SLL (Kernelsoul Logic Language) - compiled by a recursive-descent parser into executable rule sets:

WHEN hp < 30:
    IF has_potion:
        USE potion
        THEN "She uncorks a potion and drinks it in one gulp."
        BECAUSE "Character uses healing items when in danger"
    ELSE:
        THEN "Her teeth clench as blood seeps through her fingers."
        BECAUSE "Without resources, the character toughs it out"

The rule engine checks all WHEN conditions automatically after each turn. Matches are executed deterministically - no reliance on probabilistic LLM understanding.

3. Semantic Narrative Rendering

Instead of stuffing "mood: -2, energy: 8" into the prompt, the semantic renderer transforms numeric state into narrative descriptions - letting the LLM feel "atmosphere" rather than read "numbers."

ModeBehaviorUse Case
directRaw numbers: HP: 35/150Debug / minimal mode
semanticNarrative: condition fair but showing fatigueImmersive RP
hybridNumbers + narrativeBoth readability & precision

Under the hood: semantic axes, Warp functions (non-linear mapping), Gram matrices (axis decoupling), and SLERP interpolation.

4. 3-Tier Memory

5. Decision Traceability

WHY character action:
  → Rule: WHEN mood < -3 → hostile attitude
  → State: mood -5 (HP loss + repeated frustration)
  → Trigger: player provocation again
  → Result: entered hostile mode, drew sword

This is a unique Kernelsoul capability — no competitor offers traceable character behavior reasons.
